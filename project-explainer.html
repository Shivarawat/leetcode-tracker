What It Does

Problem Tracking

Add problems with difficulty, topic, DS/Algo used, and a direct link. Supports both ID and partial name lookup — type lct show "two sum" instead of remembering the number.

Approaches & Complexity

Attach multiple approaches to each problem, each with a name, time complexity, space complexity, solution code, and a note. Opens your editor for multi-line solutions.

Spaced Repetition Review

Track when you last revisited each problem. lct review shows everything not reviewed in 7+ days — so you don't forget what you've solved.

Topic Digest

lct digest graphs aggregates all takeaways, approaches used, and comments for a topic into one view. Export to Markdown for revision sheets.

Google Sheets Import

Export your existing DSA sheet as CSV and run lct import sheet.csv. Maps columns automatically — approaches, TC, SC, takeaways all come through.

Stats & Streak

See total problems by difficulty, breakdown by topic and DS/Algo, and your current solving streak — all from lct stats.

Architecture

Project Structure

  • index.js — CLI entry, all command definitions
  • src/store.js — read/write JSON data file
  • src/resolveQuery.js — ID or name lookup with picker
  • src/display.js — colored terminal output
  • src/commands/ — one file per command

Data Storage

All data stored locally at ~/.leetcode-tracker/data.json. No database, no account, no internet required. JSON is human-readable and easy to back up.

Key Technical Decisions

Commander.js for CLI

Commander is the standard Node.js CLI framework. It handles argument parsing, option flags, help text, and subcommands — so the code focuses on logic, not parsing.

JSON over SQLite

With ~500 records max, JSON is fast enough. It's human-readable, portable, easy to back up, and requires zero setup. SQLite would add complexity with no real benefit at this scale.

resolve() for lookups

Instead of forcing users to remember problem IDs, resolve() checks by ID first, then falls back to partial title match. If ambiguous, Inquirer.js shows a picker — no crash, no frustration.

~/.leetcode-tracker storage

Storing in the home directory means data persists globally — switching projects or terminals doesn't affect it. This is the same convention used by tools like ~/.ssh and ~/.gitconfig.

Inquirer for approach input

The approach command uses an interactive prompt with an editor type — it opens your default text editor for multi-line solution input, same as git commit.

Published to npm

Publishing to npm means anyone can install with npm install -g leetcode-tracker. This is the deployment equivalent for CLI tools — it makes the project real and usable by others.

Interview Q&A Prep

Q: Why did you build this?
I was already maintaining a Google Sheet to track DSA problems but it was slow to search and couldn't store my solution code or notes. I wanted something I could use from the terminal while solving problems — so I built it myself and published it to npm so others can use it too.
Q: Why a CLI instead of a web app?
When I'm solving problems I'm already in the terminal. A CLI fits the workflow — I can run lct add right after finishing a problem without switching to a browser. It also taught me how real developer tools like git and npm are structured internally.
Q: How does the name lookup work?
The resolve() function in store.js first checks for an exact ID match. If that fails, it does a case-insensitive partial title search across all problems. If one match is found it returns it directly. If multiple match, Inquirer.js shows an interactive list picker — so the user never gets a silent wrong result or a crash.
Q: How would you scale this or improve it?
A few things: swap JSON for SQLite once the dataset grows (better querying, atomic writes), add a sync command to push data to a GitHub Gist for backup, build a TUI (terminal UI) using blessed or ink for a richer interactive experience, and add LeetCode API integration to auto-populate problem metadata instead of typing it manually.
